Local AI Stack — Complete Installation Guide

An 18-phase guide to building a fully local, GPU-accelerated AI stack on Ubuntu 24.04 LTS. No cloud. No API bills. No data leaving the network.

Phase 1 — Prepare Windows for Dual Boot

Because Ubuntu will be installed on a completely separate second M.2 drive, you do not need to resize or modify your Windows C: drive at all. Windows remains fully independent.

Check second M.2 is detected in BIOS

  • Restart PC and press Delete to enter the Gigabyte BIOS
  • Go to Settings → Miscellaneous → NVMe Configuration
  • Confirm both M.2 drives are listed — the second drive should show its full capacity
  • Press Escape to return to main BIOS screen — do not exit yet

Disable Secure Boot

  • In BIOS: Settings → Miscellaneous → Secure Boot → set to Disabled

Note: Most Ubuntu 24.04 installs work with Secure Boot on, but disabling it avoids complications with ROCm and third-party drivers.

Disable Fast Startup in Windows

  • Open Control Panel → Power Options → Choose what the power buttons do
  • Click Change settings that are currently unavailable
  • Uncheck Turn on fast startup → Save changes

Note: Fast Startup keeps Windows drives in a locked state. Disabling it ensures Ubuntu can safely read your Windows drive if needed.

Press F10 to save BIOS changes and exit.

Phase 2 — Create Ubuntu USB Installer

You need a USB drive of at least 8 GB for this step.

Download Ubuntu 24.04.4 LTS

  • Go to https://ubuntu.com/download/desktop
  • Download Ubuntu 24.04.4 LTS ISO (~5 GB)

Important: Download 24.04.4 specifically — earlier point releases (24.04.1, 24.04.2, 24.04.3) ship with older kernels that do not fully support the RX 9070 XT. Verify the ISO filename includes 24.04.4 before writing to USB.

Download and Run Rufus

  • Go to https://rufus.ie and download the latest version
  • Plug in your USB drive and open Rufus
  • Select your USB drive at the top
  • Click SELECT and choose the Ubuntu ISO
  • Partition scheme: GPT
  • Target system: UEFI (non CSM)
  • Click START → choose Write in ISO Image mode when prompted

Phase 3 — Install Ubuntu 24.04 LTS on Second M.2

Boot from USB

  • With USB plugged in, restart PC
  • Press F12 at the Gigabyte splash screen to open boot menu
  • Select your USB drive

Run the Installer

  • Select Try or Install Ubuntu
  • Double-click Install Ubuntu on the desktop
  • Language: English (or your preference)
  • Keyboard layout: match your keyboard

Partitioning — Critical Step

Warning: On the installation type screen, choose Something else (manual partitioning). Do NOT choose "Install Ubuntu alongside Windows" — that targets your C: drive.

  • Select Something else → Continue
  • In the partition table, identify your second M.2 drive — it will show as /dev/nvme1n1 with its full capacity available
  • If the second drive has old partitions, select each one and click the minus (−) button to delete them all
  • Select the free space and click the plus (+) button to create a new partition:
SettingValue
SizeAll remaining free space (leave at maximum)
Type for the new partitionPrimary
Location for the new partitionBeginning of this space
Use asExt4 journaling file system
Mount point/
  • At the bottom of the screen, find Device for boot loader installation
  • Set this to your second M.2 drive (e.g. /dev/nvme1n1) — NOT the Windows drive (/dev/nvme0n1)
  • This installs GRUB on the Linux drive, leaving your Windows drive completely untouched
  • Click Install Now → confirm → continue with timezone and account setup
  • Timezone: Australia/Sydney
  • Create your username and password
  • Wait for install to complete (~10–20 mins) → remove USB when prompted → reboot

Set boot drive order in BIOS

  • After reboot, press Delete to enter BIOS
  • Go to Boot → Boot Option Priorities
  • Set Boot Option 1 to your second M.2 (the Linux drive)
  • Set Boot Option 2 to your first M.2 (the Windows drive)
  • Press F10 to save and exit
  • GRUB will now appear on every startup with Ubuntu and Windows Boot Manager as options

Note: To boot Windows without the GRUB menu (e.g. for gaming), press F12 at startup and select the Windows drive directly. You can also change the default boot OS and countdown timer later using the grub-customizer app in Ubuntu.

Phase 4 — Post-Install Ubuntu Setup

Update the system

sudo apt update && sudo apt upgrade -y

Fix dual-boot clock conflict

timedatectl set-local-rtc 1

Install AMD GPU Drivers and ROCm

ROCm enables your RX 9070 XT to accelerate AI inference on Linux — this is the main advantage of dual booting.

# Download and install the amdgpu-install tool for Ubuntu 24.04.4 (Noble)
# ROCm 7.2.1 — current stable with confirmed RX 9070 XT Noble support (March 2026)
# Note: 7.2.0 had a missing Noble repository; 7.2.1 resolves this.
# To check for newer releases: https://repo.radeon.com/amdgpu-install/
sudo apt update
wget https://repo.radeon.com/amdgpu-install/7.2.1/ubuntu/noble/amdgpu-install_7.2.1.70201-1_all.deb
sudo apt install -y ./amdgpu-install_7.2.1.70201-1_all.deb

# Install ROCm and graphics usecase
# --usecase=graphics,rocm is current AMD-recommended syntax for desktop Radeon installs
# --no-dkms removed: no longer recommended for desktop systems with a connected display
sudo amdgpu-install -y --usecase=graphics,rocm

# Add your user to the required groups
sudo usermod -a -G render,video $USER
sudo reboot

Note: The installer script pins version 7.2.1 for reliability. Using the version-pinned .deb avoids picking up untested newer releases from the apt repo. ROCm 7.2.0 had a missing Noble repository — 7.2.1 (March 2026) resolves this and is the current recommended version for the RX 9070 XT on Ubuntu 24.04.4.

After reboot, verify GPU is detected:

rocminfo | grep gfx

Optional but recommended: Install the HWE kernel for the best RX 9070 XT hardware support (kernel 6.17 adds RDNA4 fan speed reporting and additional hardware features):

sudo apt install linux-generic-hwe-24.04
sudo reboot

Install essential tools

sudo apt install -y curl git wget gnupg ca-certificates \
    python3 python3-pip ffmpeg build-essential \
    nodejs npm openssl lsb-release \
    apt-transport-https software-properties-common

Phase 5 — Install Docker

All AI services in this guide run as Docker containers on Ubuntu. Install Docker Engine (not Docker Desktop):

sudo apt install -y ca-certificates curl gnupg
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
    | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
    https://download.docker.com/linux/ubuntu \
    $(. /etc/os-release && echo $VERSION_CODENAME) stable" \
    | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update && sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
sudo usermod -aG docker $USER
newgrp docker
docker --version

Note: Adding your user to the docker group means you can run docker commands without sudo. The newgrp command applies this immediately without needing to log out. A full reboot is still recommended.

Phase 6 — Install Ollama

Ollama runs natively on Ubuntu and automatically uses your RX 9070 XT via ROCm for GPU-accelerated inference — significantly faster than CPU-only operation.

curl -fsSL https://ollama.com/install.sh | sh
ollama --version
systemctl status ollama
  • Verify Ollama is running by visiting http://localhost:11434 in a browser — you should see: Ollama is running

Note: Ollama installs as a systemd service and starts automatically on boot. If ROCm was set up correctly in Phase 4, Ollama will detect and use the RX 9070 XT automatically.

Phase 7 — Install Open WebUI

Open WebUI runs as a Docker container and provides the browser-based chat interface for all your local models.

docker run -d -p 3000:8080 \
    --add-host=host.docker.internal:host-gateway \
    -v open-webui:/app/backend/data \
    --name open-webui --restart always \
    ghcr.io/open-webui/open-webui:main
  • Wait for Docker to pull and start the image — this takes a few minutes on first run
  • Open a browser and go to http://localhost:3000
  • Create your admin account on first launch — this stays fully local
  • Go to Settings → Connections and confirm Ollama URL shows http://host.docker.internal:11434

Phase 8 — Pull Gemma 4 Model

Gemma 4 is Google DeepMind's latest open model family. All variants support text and image input and include configurable reasoning/thinking modes. Choose based on your VRAM:

ModelVRAMContextArchitectureNotes
gemma4:e2b~7 GB128KDense (edge)Fast, good for testing
gemma4:e4b~10 GB128KDense (edge)Recommended — best balance
gemma4:26b~16 GB256KMoE (3.8B active/token)Near-max quality — tight on 16GB cards

Recommended — e4b model

ollama pull gemma4:e4b

Alternative — e2b model (faster, lighter)

ollama pull gemma4:e2b

Large — 26b model (near-maximum quality)

ollama pull gemma4:26b

Warning: The 26b model needs ~16GB VRAM — your RX 9070 XT's full capacity. Long conversations build up a KV cache that can spill to system RAM and slow inference. Monitor VRAM usage with: rocm-smi

Pull the embedding model (required for OpenBrain and Obsidian)

# nomic-embed-text is used by OpenBrain and Obsidian Smart Connections
ollama pull nomic-embed-text

Test the model is working

ollama run gemma4:e4b

Note: The model will appear automatically in the Open WebUI model selector.

Phase 11 — Install yt-dlp (YouTube & Web Video)

yt-dlp downloads audio from YouTube and other video platforms for transcription with Whisper. It can also pull existing subtitles and transcripts directly without downloading the full video.

Install on Ubuntu

sudo pip install yt-dlp --break-system-packages
yt-dlp --version

Download audio only (send to Whisper)

yt-dlp -x --audio-format mp3 -o "%(title)s.%(ext)s" "https://youtube.com/watch?v=VIDEO_ID"

Download transcript directly (no audio needed)

yt-dlp --write-auto-sub --skip-download --sub-format txt -o "%(title)s" "https://youtube.com/watch?v=VIDEO_ID"

Note: The direct transcript option only works if YouTube has auto-generated subtitles. If not, download the audio and use Whisper instead.

Batch process a playlist

yt-dlp -x --audio-format mp3 -o "%(playlist_index)s - %(title)s.%(ext)s" "https://youtube.com/playlist?list=PLAYLIST_ID"

Best practices for knowledge base ingestion

  • Scanned PDFs — upload directly, Tika will OCR them automatically
  • Spreadsheets — for large datasets, export as CSV first for better RAG retrieval

  • Upload files individually for large batches — prevents timeout errors during processing
  • After upload, check for a green tick on each file confirming successful processing

Phase 14 — Remote Collaboration with Tailscale

Tailscale creates a private encrypted VPN between your machine and your colleague's machine. They can access your Open WebUI as if they were on your local network — without exposing anything to the public internet. Free for up to 6 users with unlimited devices.

Without TailscaleWith Tailscale
Open WebUI only accessible at localhost:3000Colleague accesses via a private URL from anywhere
Sharing requires opening your router to the internetNothing exposed to the internet — fully private VPN
No authentication on the network levelWireGuard encryption, Tailscale account required to join
Colleague needs to be on your local networkWorks from anywhere — home, office, cafe

Step 1 — Create a Tailscale account
Go to https://tailscale.com and sign up for a free account. The free plan supports up to 6 users with unlimited devices.

Step 2 — Install Tailscale on Ubuntu

curl -fsSL https://tailscale.com/install.sh | sh

A URL will appear in the terminal — open it in a browser to authenticate your machine.

Go to https://login.tailscale.com/admin/dns → Under HTTPS Certificates, click Enable HTTPS, then run:

sudo tailscale cert your-machine-name.tail1234.ts.net

Step 4 — Invite your colleague
Go to https://login.tailscale.com/admin/users → click Invite users and enter your colleague's email address.

Step 5 — Share your machine
In the Tailscale admin console, go to Machines → find your Ubuntu machine → click the three dots menu → Share → enter your colleague's Tailscale email.

Step 6 — Create your colleague's Open WebUI account
In Open WebUI: Admin Panel → Users → New User. Set their role to User — you remain the admin.

Phase 15 — Install OpenBrain (Persistent AI Memory)

OpenBrain creates a single persistent memory database that any AI tool can read from and write to — so your context travels with you across sessions and across different AI tools. Everything stays on your Ubuntu machine.

Without OpenBrainWith OpenBrain
Every conversation starts blankAI recalls your projects, preferences, and past decisions automatically
Re-explain context every sessionContext is written once and retrieved by meaning across all sessions
Memory locked to one AI platformOne shared database — Claude, Open WebUI, ChatGPT all read the same memory
Data stored in company serversFully local — all data stays on your Ubuntu machine
Keyword-only searchSemantic search — 'career change' finds 'Sarah is leaving her job'

Architecture

  • PostgreSQL + pgvector — stores memories as both raw text and vector embeddings for semantic search
  • MCP server — bridges the database to any MCP-compatible AI tool
  • Ollama embeddings — your local Ollama instance generates vector embeddings, keeping everything on-device
  • REST API — lets you capture memories from scripts, Slack, or any other source

Phase 16 — Install Obsidian (Knowledge Base + Local AI)

Obsidian is a local-first Markdown note-taking app that stores everything as plain text files on your machine — no cloud sync required. Combined with your local Ollama stack, it becomes a private knowledge base where your AI can chat with your notes.

Without Obsidian + AIWith Obsidian + AI
Notes are static — searchable by keyword onlySemantic search finds notes by meaning, not just words
AI has no access to your personal knowledge baseAI answers questions grounded in your actual notes
Context lost between AI sessionsYour vault is persistent — knowledge accumulates over time
Notes may sync to the cloudEverything stays local — plain Markdown files you own

Step 1 — Install Obsidian via official .deb

# Download the latest Obsidian .deb from GitHub releases

  • Launch Obsidian from the application menu or run: obsidian
  • On first launch, create a new vault — choose a folder on your Linux drive (e.g. ~/ObsidianVault)

Note: Obsidian requires a graphical desktop session — it cannot run headlessly.

Step 2 — Enable Community Plugins

  • Open Obsidian → Settings (gear icon, bottom left)
  • Go to Community plugins
  • Click Turn on community plugins → confirm
  • Click Browse to open the plugin marketplace

Step 3 — Install Smart Connections (semantic vault search)

Smart Connections embeds every note in your vault using nomic-embed-text — the same embedding model already running in your Ollama stack.

  • Search for Smart Connections → Install → Enable
  • Go to Settings → Smart Connections
  • Set Embedding Model to: Ollama
  • Set Ollama URL to: http://localhost:11434
  • Set Embedding model name to: nomic-embed-text
  • Set Chat model to: gemma4:e4b
  • Click Force Refresh — Smart Connections will now embed your entire vault

  • Search for Copilot → Install → Enable

  • Set Default model to: gemma4:e4b

Recommended vault structure for your AI stack

~/ObsidianVault/
├── Daily/          # Daily notes — date-stamped, linked to projects
├── Projects/       # One folder per project, with a MOC (map of content) note
├── AI-Sessions/    # Saved outputs and summaries from Open WebUI chats

Phase 17 — Install context-mode (Context Window Optimiser)

context-mode is an MCP server that sits between Claude and its tools. When a tool call would dump large output into the conversation, context-mode intercepts it and runs it in a sandboxed subprocess instead — the raw data never touches the context window.

Without context-modeWith context-mode
A Playwright snapshot costs 56 KB of contextSame snapshot costs 299 bytes — 99% reduction
Context window exhausted after ~30 minutesSessions run for ~3 hours before degradation
After compaction, Claude forgets files, tasks, decisionsFull session state recovered — continues from your last prompt
Reading 50 files = 50 tool calls, 700 KB of contextOne script analyses all 50 files, logs only the result — 3.6 KB

Important: context-mode works with Claude Code, claude.ai, VS Code Copilot, Cursor, and similar AI coding agents. It has no effect on Open WebUI or local Gemma 4 sessions.

How it fits alongside OpenBrain and Obsidian

ToolWhat it handlesScope
context-modeRaw tool output — keeps code execution, file reads, and API responses out of contextWithin a session
OpenBrainDecisions, preferences, project context that persists between conversationsAcross sessions
ObsidianStructured notes, project documentation, session summariesPermanent knowledge base

Step 1 — Install context-mode globally

npm install -g context-mode

# Verify installation
context-mode --version
context-mode doctor

Step 2 — Copy AGENTS.md routing instructions

cp $(npm root -g)/context-mode/configs/claude-code/CLAUDE.md ~/AGENTS.md

Note: OpenCode now prioritises AGENTS.md over CLAUDE.md — if both exist, only AGENTS.md is used. The file is copied from the same source but saved as AGENTS.md. Claude Code continues to use CLAUDE.md.

Step 3 — Add to Claude Code

# MCP-only install (simplest — all 11 tools, no automatic hook routing)
claude mcp add context-mode -- npx -y context-mode

Phase 18 — Install OpenCode (Terminal Coding Agent)

OpenCode is a free, open source AI coding agent available as a terminal TUI, IDE extension for VS Code and Cursor, and desktop app — similar to Claude Code but not tied to any single provider. It can run against your local Gemma 4 via Ollama at zero cost and with full privacy.

Step 1 — Install OpenCode

The recommended install method is the official script:

curl -fsSL https://opencode.ai/install | bash
opencode --version

Alternatively, install via npm:

npm install -g opencode-ai@latest
opencode --version

Step 2 — Configure local Gemma 4 as default model

# Add to ~/.bashrc to make permanent
echo 'export LOCAL_ENDPOINT=http://localhost:11434' >> ~/.bashrc
source ~/.bashrc

Step 3 — context-mode is pre-wired

The installer writes OpenCode's global config with context-mode already registered:

{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "context-mode": {
      "type": "local",
      "command": ["context-mode"],
      "enabled": true
    }
  }
}

Note: The "plugin" key has been removed from the OpenCode config schema. MCP servers are now activated via "enabled": true within the "mcp" block.

Step 4 — Copy AGENTS.md to each new project

# Template saved by the installer — copy to each project root
cp ~/ai-stack/opencode_AGENTS.md.template ~/your-project/AGENTS.md

# Or let OpenCode generate it automatically on first run
cd ~/your-project
opencode
# Then type: /init

Step 5 — Start coding

cd ~/MarketSimulator
opencode

# Verify context-mode is working — type this in the OpenCode session
ctx stats

OpenCode has two modes you switch between with Tab:

  • Plan mode — OpenCode drafts what it intends to do before touching any files. Review the plan, give feedback.
  • Build mode — OpenCode executes the plan, edits files, runs commands, checks results.

Note: The recommended way to authenticate is opencode auth login — select Anthropic and sign in with your Claude Pro or Max subscription. This is the most cost-effective approach. If you use a raw Anthropic API key instead, every session uses API credits — check anthropic.com/pricing before heavy sessions. For completely cost-free coding, use local Gemma 4 via LOCAL_ENDPOINT.
