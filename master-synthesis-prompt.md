## 3. Content Philosophy

### My Ideas
Original concepts, business ideas, and thinking — some decades in the making. Each entry includes:
- The idea itself
- When it first emerged (approximate)
- Current status / development stage
- Connected Skills and Signals (via tags)

### Signals
External influences — people and ideas that have shaped Cameron's thinking. Signals come in two types:

**Type 1 — Reference Signals** *(linked, sourced)*
Where the original exists and can be credited. URL, document, person, publication.

**Type 2 — Influence Signals** *(attributed, remembered)*
Where the source shaped Cameron but may be gone, paraphrased, or simply a person whose thinking permeated his over time. No URL required. Honest attribution without pretending a source exists when it doesn't.

Each Signal includes:
- Source (person, publication, institution, or AI session)
- Link (if available)
- Why it matters — in Cameron's own voice
- Tags connecting to Skills and/or My Ideas

**Note:** Claude is a named Signal source. AI collaboration sessions that produce genuine insight are attributed the same way any significant intellectual influence would be. Session 1 (17 May 2026) should be the first Signal entry.

### Skills
Honest Markdown files — not progress bars or percentages. Each skill file follows this template:

```markdown
## What this skill means to me
## Where it showed up in my work
## What I'm still figuring out
## Ideas or Signals connected to this skill
```

## 4. Radical Collaboration Transparency (RCT)

**First implementation:** cameronloudon.com  
**Date coined and implemented:** 17 May 2026  
**Originator:** Cameron Loudon  
**Co-developed with:** Claude (Anthropic) — claude-sonnet-4-6  
**License:** CC BY 4.0

### What It Is

Radical Collaboration Transparency is a personal publishing framework based on one principle:

> *AI collaboration is an intellectual influence like any other — and should be attributed honestly, not hidden or over-explained.*

Most people either hide AI involvement or over-disclaim it. RCT does neither. It treats AI the same way a thoughtful person treats any significant intellectual influence — a mentor, a book, a conversation that changed your thinking. You name it. You credit it. You show your work.

### What It Means in Practice

Every piece of content on cameronloudon.com that emerged from or was shaped by an AI session carries a **Collaboration Note** — a footer block that states full provenance and telemetry. Cameron values telemetry and defaults to more detail rather than less. Granularity can be adjusted per session, but the default is full transparency:

```
🤝 Collaboration Note
This content was developed in a thinking session with Claude (Anthropic).
Model: claude-sonnet-4-6
Session date: [date]
Approximate tokens used: [number]
Session duration: [time]
Output reviewed and approved by Cameron Loudon.
```

### What It Is Not
- Not about humanising AI output to disguise it
- Not performative AI disclosure for compliance
- Not a claim that AI wrote everything
- A genuine practice of showing how Cameron thinks and who — and what — he thinks with

## 5. The Co-Collaborator Model — How Sessions Work

### Session Workflow

```
Cameron + Claude (or other AI) have a thinking session
→ Ideas, Signals, Skills, or Now page content emerges
→ Claude produces a session summary in Markdown
→ Cameron reviews and approves
→ Approved content published to GitHub Pages
→ Each published piece carries a Collaboration Note
```

**Claude is a named Signal source.** When a session produces genuine insight, it is attributed in the Signals section the same way any significant intellectual influence would be.

### Branch Workflow (from Session 2 onwards)
This is the technical implementation of the RCT review step:

```
main branch     → live site (always protected, never worked on directly)
session-N branch → where Claude Code does all work

Workflow:
1. Claude Code creates branch: session-2 (or session-3, etc.)
2. All file changes happen on the session branch
3. Session ends → Claude produces summary
4. Cameron reviews the diff on GitHub (visual, line by line)
5. Cameron merges to main → site goes live
6. Session branch deleted → clean slate for next session
```

If anything goes wrong on the session branch — delete it. Main is untouched.

**Branch naming convention:** `session-[number]` e.g. `session-2`, `session-3`

Cameron's approach to Claude Code:
- Always work on a branch, never directly on main
- Keep "Accept all" mode off until confident in what Claude Code is doing
- Use GitHub web interface to review commit history visually after each session

### What to Produce at the End of Each Session

1. **Session Summary** — what was discussed, decided, and created
2. **Ready-to-publish Markdown/HTML files** — formatted for Jekyll
3. **Collaboration Note** — model, date, approximate tokens, duration
4. **Updated Now page draft** — if relevant
5. **New Ideas, Signals, or Skill file drafts** — ready for the Jekyll folder structure
6. **Updated Master Synthesis Prompt** — suggested additions/changes so it stays current

### Jekyll Frontmatter Template
